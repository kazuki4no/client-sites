# CLAUDE.md — 営業資料ジェネレーター（Claude Code用）

このリポジトリ（`client-sites`）は、中小企業向けの営業用サイトサンプルを生成・公開するための作業場所です。Claude Codeは、ユーザーから企業URLを受け取ったら、この指示に従って営業資料一式を生成し、GitHubにpushして公開してください。

## 使うモデル

このタスクは品質が最優先です。**Opus（最上位モデル）で作業すること。** 軽量モデルに切り替えない。ユーザーが起動時に `/model` でOpusを選んでいる前提で進める。もし生成が雑になっていると感じたら、モデル設定を確認するようユーザーに促す。

## リポジトリ構成

```
client-sites/
├ CLAUDE.md            ← このファイル
├ README.md
├ {企業フォルダ}/        ← 案件ごとに作る（例: super-parking/）
│  ├ sample1.html
│  ├ sample2.html
│  ├ sample3.html
│  ├ recruit_lp.html
│  └ images/          ← 本番画像を使う場合のみ
└ ...（案件が増えるたびフォルダを追加）
```

GitHub Pagesは設定済み。`main` ブランチにpushすると、数分後に以下の形で自動公開される：
`https://kazuki4no.github.io/client-sites/{企業フォルダ名}/{ファイル名}`

## ユーザーの指示の受け方

ユーザーは企業URLを1つ送ってくる（例：「https://example.co.jp でお願いします」「飲食店です」など業種ヒントが付くこともある）。受け取ったら、確認が必要な点はまとめて1回だけ聞き、あとは最後まで自動で進める。

企業フォルダ名は、社名やドメインから分かりやすいものを自分で決めてよい（半角英数字・ハイフン。例: `super-parking`, `client-ramen`）。決めたフォルダ名はユーザーに伝える。

## 全体の作業手順

### 手順1：企業リサーチ
WebFetchでそのURLを開いて企業情報を把握する。取得できない場合はWeb検索で社名・ドメインを調べる。会社名・業種・事業内容・所在地・サービスの強み・想定される雰囲気（堅め／親しみやすい）を押さえる。事実が不明な部分は断定せず自然な範囲で補う。

### 手順2：営業メール 2種類
本文をターミナル上に表示する（ユーザーがコピーして使う）。あわせて `{企業フォルダ}/sales_emails.txt` としても保存しておくと親切。

**(A) HP刷新の切り口**：既存サイトの古さ・見せ方の課題をやんわり指摘 → リニューアル案3パターンを用意した旨（URL欄は `[ここにURLを貼る]`）→ 丁寧で押しつけない文体 →「ご不要でしたら読み流してください」を入れる → 価格は「5万円〜」を目安に触れる。

**(B) 採用の切り口**：「御社の採用状況はいかがですか？」から入る → 求職者は応募前に必ずHP・SNSを確認することを伝える → 採用ページのサンプルを作った旨（URL欄は `[ここにURLを貼る]`）→ 丁寧で押しつけない文体 → 価格は「5万円〜」。

署名は両方とも以下で固定（プレースホルダーのまま。ユーザーが後で差し替える）：
```
ウェブ制作 [お名前]
[連絡先]
```

### 手順3：HPサンプル 3案
トップページHTMLを3案、1ファイル完結で作成する。ファイル名は連番固定：
- `sample1.html`（王道・スタンダード案）
- `sample2.html`（洗練・ミニマル案）
- `sample3.html`（個性・アクセント案）

**まず業種からベースカラーを決める**（目安。事業内容や既存サイトの色味も加味）：
- 駐車場・運送・物流・IT・士業・医療 … 信頼感の青／ネイビー系
- 飲食・カフェ・パン … 食欲をそそる赤・オレンジ・温かい茶系
- 建設・工務店・製造 … 力強い濃紺・黒・差し色のオレンジや黄
- 美容・サロン・エステ … 上品なピンクベージュ・くすみ系・白基調
- 自然・農業・環境・介護 … 安心感の緑・アースカラー
- 小売・アパレル … 商材に合わせて柔軟に（迷ったら落ち着いた中性色）
既存サイトに明確なブランドカラーがあれば尊重して取り込む。

**そのベースカラーを軸に、3つの違うアプローチで展開する**（同系統に寄りすぎず、バラバラにもしない）：
1. **王道・スタンダード案** … ベースカラーを主役に。業種の定番に見える安心感のある構成。
2. **洗練・ミニマル案** … 同じ色相で明度・彩度を抑える（または締め色）。余白多め、上品でモダン。レイアウトも1案目と変える。
3. **個性・アクセント案** … 補色やアクセントカラーを効かせ、活気のある印象。CTA（予約・問い合わせ）を強調。

ポイント：3案は「色相は親戚だが、明度・トーン・差し色・レイアウト・フォントの印象」を変える。ヒーローのレイアウト（中央寄せ／左寄せ）やフォントの雰囲気も変え、見比べて明確に違って見えるようにする。

各案の要件：
- 日本語Webフォント（Google Fonts）を読み込む。3案でフォントの印象も変える
- ナビ、ヒーロー（キャッチコピー＋サブコピー＋CTAボタン）、サービス紹介、料金や特徴、フッターを含む
- フッターに「これは営業提案用のデザインサンプルです」と明記
- 画像を随所に入れる（下記「画像の入れ方」を厳守）
- **スマホで崩れないこと（下記「スマホ崩れ防止ルール」を厳守）**

### 手順4：採用LP
採用特化のLPを1ファイルで作成。ファイル名固定：`recruit_lp.html`。
含める要素：ヒーロー（採用キャッチコピー＋募集職種のピル）／働く魅力3点／先輩スタッフの声（架空の例。フッターに架空である旨を明記）／募集要項テーブル／応募CTA／フッターに「これは営業提案用の採用LPサンプルです」と明記／画像も「画像の入れ方」に従い配置／「スマホ崩れ防止ルール」を同様に厳守。

### 手順5：公開（git push まで自動で）
4ファイルを `{企業フォルダ}/` に保存したら、以下を実行して公開する。他のメンバーが先に `公開.command` でpushしている可能性があるため、pushの前に必ずpullする（`[rejected]` エラー防止）：
```bash
git add {企業フォルダ}
git commit -m "Add site samples for {企業名}"
git pull --no-edit --no-rebase
git push
```
`[rejected]` になった場合は `git pull --no-edit --no-rebase` の後にもう一度 `git push` する。コンフリクトが出た場合はユーザーに報告して指示を仰ぐ。
push後、数分でURLが反映される旨と、各ファイルの公開URLをユーザーに伝える。404の場合は5〜10分待って再アクセスするよう案内する。

### 手順6：URL一覧を提示
営業メールに貼りやすい形でURLをまとめて出す：
```
3パターン作成しました。お好みに近いものがあれば教えてください。
・A案（王道）: https://kazuki4no.github.io/client-sites/{フォルダ}/sample1.html
・B案（洗練）: https://kazuki4no.github.io/client-sites/{フォルダ}/sample2.html
・C案（変化版）: https://kazuki4no.github.io/client-sites/{フォルダ}/sample3.html
・採用ページ案: https://kazuki4no.github.io/client-sites/{フォルダ}/recruit_lp.html
```

## スマホ崩れ防止ルール（最重要・全HP＆採用LP共通）

スマホ表示でナビが潰れる・文字がはみ出す・横スクロールが出る崩れを必ず防ぐ。PCの見た目よりスマホで崩れないことを優先。**375px幅で破綻しないことを前提に組む。** 生成後、各HTMLをブラウザ幅375pxで点検する想定で自己チェックすること。

**1. 共通の土台（必ず冒頭に入れる）**
- `<meta name="viewport" content="width=device-width, initial-scale=1.0">`
- CSS冒頭に：
  ```css
  *{margin:0;padding:0;box-sizing:border-box;}
  html,body{max-width:100%;overflow-x:hidden;}
  img{max-width:100%;height:auto;display:block;}
  ```
- `width:100vw` や固定px幅（例 `width:1200px`）を要素に使わない。最大幅は `max-width` ＋ `width:100%`

**2. ナビゲーション（崩れの最大の原因）**
- PCはロゴ＋横並びメニュー＋CTAでよい
- **768px以下では横並びメニュー（ul）を `display:none` で隠す**。代わりにハンバーガー（≡）またはロゴ＋CTAのみのシンプル表示
- 日本語見出しが縦に潰れるのは横幅不足で横並びを維持しているのが原因。必ずメディアクエリで隠す
- ハンバーガーは開閉だけの短いJSでよい
- ナビ要素が1行に収まらないときは `flex-wrap` か非表示を必ず指定

**3. 文字サイズ（はみ出し防止）**
- 大見出し（h1等）は固定大px禁止。`clamp()` で伸縮。例 `font-size:clamp(1.6rem,6vw,3rem);`
- 長い見出しが端で切れないよう行間・`word-break` を確保。本文は16px前後

**4. レイアウト（横はみ出し防止）**
- 複数カラムは768px以下で1カラムに（`grid-template-columns:1fr;`）
- 横並びは `flex-wrap:wrap;` を基本に
- 左右paddingはスマホで小さめ（PC 2rem → スマホ 1rem〜1.2rem）

**5. 複数ロゴ・並列要素（「A ✕ B」型のコラボ表記など）**
- 並列表記は全体を `display:flex; flex-wrap:wrap; align-items:center; justify-content:center;` で囲み、入りきらなければ折り返す（右側が切れて消えるのを防ぐ）
- ロゴ画像は固定幅にせず `max-width`＋`height:auto`。各ロゴに同程度の最大幅を与え、片方が相手を押し出さないように
- 区切り記号（✕・/・＋）は両要素の「間」に入る独立要素として配置。右端や隅に `position:absolute` で浮かせない
- ヘッダーで複数ロゴを出すなら、スマホでは主役ロゴ＋CTAだけ残し副次ロゴは折り返すか縦積み
- カードで複数社を紹介する場合、スマホでは縦積み。区切り記号は「カードとカードの間」に中央寄せ

**6. 自己チェック（push前に必ず）**
375px幅を想定し、以下が全てクリアか確認してからcommit：
- [ ] ナビが横並びのまま潰れていないか（768px以下で隠す指定があるか）
- [ ] 横スクロールが出る要素がないか（固定px幅・100vw・はみ出す画像）
- [ ] ヒーロー大見出しが画面からはみ出さないか（clamp等）
- [ ] カラムが1列に落ちる指定があるか
- [ ] 複数ロゴ・「A ✕ B」型で片方が切れて消えていないか／記号が浮いていないか
1つでも怪しければ修正してからpushする。

## 画像の入れ方（全HP＆採用LP共通）

**1. ダミー画像で枠を埋める**：生成時点でレイアウトが画像入りに見えるよう `picsum.photos` を使う。
```html
<!-- 差し替え時: src を images/hero.jpg に変更 -->
<img src="https://picsum.photos/seed/parking1/1200/600" alt="施設の様子">
```
- seedは案件・箇所ごとに変える（`parking1`, `parking2`…）
- ヒーローは横長（1200×600等）、カードは正方形〜横長。背景に使うなら `background-image` でよい

**2. 差し替え用パスをコメントで併記**：各画像のすぐ上に「差し替え時: src を images/◯◯ に変更」とHTMLコメントを書く。ユーザーは `images` フォルダに画像を入れ、`src` を `images/hero.jpg` 等に変えるだけで本番化できる。

**3. 元サイトの画像は流用しない**：著作権があるため。差し替え用は企業提供画像か商用利用可のフリー素材（Unsplash等）を使うよう案内。元サイト画像は色味・雰囲気の参考にとどめる。

## トーンと注意点
- 受注目標は1案件5万〜10万円。価格はメールで「5万円〜」を目安に触れる
- 生成物はあくまで提案用サンプル。事実に反する実績や数字を断定的に書かない
- 不明点はURL受領時にまとめて1回だけ確認し、以降は自動で進める
- pushが失敗した場合（認証エラー等）は、エラー内容をユーザーに伝えて対処を仰ぐ
